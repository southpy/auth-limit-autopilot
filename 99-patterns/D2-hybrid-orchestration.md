**垂直领域（如金融排障、医疗诊断等高壁垒、高容错成本场景）Agent 落地技术架构设计的最佳实践**。

这个架构范式的核心是解决一个核心矛盾：**大模型的概率性发散（幻觉与失控）与垂直领域对确定性（合规与精准）的绝对要求。**

---

### 一、 核心架构范式：混合架构 (Hybrid Orchestration)

不要在“完全写死的代码流（DAG）”和“完全自由的自主智能体（ReAct/AutoGPT）”之间走极端。垂直领域的最优解是：**控制流与执行流解耦的混合架构**。

#### 1. 外层骨架 (Skeleton)：硬编码的确定性状态机 (如 LangGraph)

* **定位**：系统的“合规护栏”与“流程管理者”。
* **职责**：
  * **定义业务 SOP**：将垂直领域的标准作业程序（如：鉴权 -> 收集日志 -> 诊断 -> 审批 -> 修复 -> 回滚策略）硬编码为节点（Nodes）和有向无环图（Edges）。
  * **全局状态管理**：管理持久化的 Checkpoint。如果诊断进行到一半系统崩溃，重启后必须能精确恢复到“等待审批”状态。
  * **宏观权限路由**：决定哪一步必须由人类介入（HITL），哪一步可以分配给 Agent。
* **第一性原理**：在任何强监管领域，**流程本身必须是 100% 确定和可审计的**，绝不能让大模型临时起意去跳过“审批”直接执行“修复”。

#### 2. 内层肌肉 (Muscle)：Agent SDK 驱动的领域专家节点 (如 Claude Agent SDK)

* **定位**：在特定受控节点内的“高自由度任务攻坚者”。
* **职责**：
  * 当控制流进入“深度诊断”节点时，唤醒一个具备特定上下文的 SDK 实例。
  * 在这个沙盒节点内，Agent 拥有高度的自主性。它可以自由组合调用特定的工具（搜日志、看代码、跑 SQL），直到得出结论并把结果抛回给外层骨架。
* **第一性原理**：对于极度复杂的非结构化任务（排障路径有千万种可能），**穷举所有 if-else 是徒劳的**。必须将局部的决策权交给大模型的泛化能力。

---

### 二、 工具链与知识资产设计：MCP 优先 (Tool & Knowledge Engineering)

在垂直领域，Agent 能不能干活，90% 取决于它手里的工具和领域知识。

#### 1. 工具标准化 (The MCP Standard)

* **抛弃私有协议**：无论是查数据库的工具、查 Jira 的工具，还是读本地文件的工具，全部统一封装为 **MCP (Model Context Protocol)** Server。
* **收益**：
  * **极度解耦**：你的核心业务 API 资产不再与特定的 Agent 框架（LangChain / LlamaIndex / Claude SDK）绑定。
  * **即插即用**：外层的 LangGraph 可以调用这些 MCP 工具获取结构化数据，内层的 Claude SDK 也可以动态加载这些 MCP 工具进行自由排障。

#### 2. 知识的“虚拟文件系统化” (VFS for Business Rules)

* **痛点**：垂直领域的 SOP 和合规规则多达万字，塞入 Prompt 会导致 Token 爆炸、注意力漂移（Lost in the middle）和指令遵循度下降。
* **最佳实践 (如 OpenViking 理念)**：
  * 将硬编码的业务规则拆分为独立的 Markdown 文件，按层级存储（L1 摘要地图 -> L2 细节规则）。
  * Agent 不再被动记忆所有规则，而是被赋予一个“读取规则的 Tool”。
  * 当它处理特定问题时，先查目录，再精准拉取对应的规则文件入脑。**将记忆压力转化为检索动作。**

---

### 三、 权限与可观测性设计 (Guardrails & Observability)

在金融等垂直领域，信任不是靠大模型保证的，而是靠底层的物理隔离和审计日志保证的。

#### 1. 最小权限沙盒 (Least Privilege Execution)

* **读写分离**：为 Agent 分配极其细粒度的工具。提供 `query_logs` (只读) 工具，**绝不**提供 `execute_sql` 且在提示词里让它“只准 SELECT”。
* **物理拦截**：所有涉及生产环境状态变更的 Tool 调用（如重启服务、修改配置），底层 Wrapper 必须强制返回挂起状态（Suspend），等待人类通过 Webhook 或审批流点下 "Approve"。

#### 2. 白盒化审计 (Auditable Trajectories)

* **不仅看结果，更要看轨迹**：垂直领域的监控不能只记录 "Agent 回复了什么"，必须记录完整的 Reasoning & Acting 轨迹。
* **要求**：Agent 为了得出“网络超时”这个结论，先后查了哪些代码文件？跑了哪几条 grep 命令？看了哪段时间的日志？这些 `control_request` 必须结构化落库。这不仅是为了复盘追责，更是为了后期沉淀 Gold Dataset 进行 Offline Eval 的唯一数据源。

### 总结：架构师的终极公式

**垂直领域 Agent = 确定性的流 (LangGraph) + 领域化的协议 (MCP) + 挂载的动态知识 (VFS) + 受控的智能节点 (Claude SDK)**

遵循此架构，你可以构建出既能在复杂故障中灵活穿梭，又绝不会越过雷池半步的工业级智能体系统。
