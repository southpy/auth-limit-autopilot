## 核心发现

### F1：排障本质是「交易→额度变化」的可解释性问题

**发现：**

每笔交易后，可用额度变化遵循公式：

```
可用额度变化 = f(交易金额，业务规则，系统参数)
```

当客户感知的变化 ≠ 预期变化时产生疑问。**排障 = 还原这个 f 的完整公式，并向客户解释差异原因。**

**问题场景分类**：

- **APP 查询限制**：客户通过 APP 仅能查询部分交易明细，对于部分内生交易（如 4029/4036 等入账交易）不可见，是客诉原因之一
- **隐性额度变动**：除消费、分期、还款、退货等客户感知的交易外，以下变动对可用额度均有影响，也可能导致客户体验不一致：
  - 银行汇率参数调整
  - 临额/固额生效失效
  - 授权交易过期
- **单笔交易差异**：交易金额 ≠ 实际账户可用额度前后变化值时，与额度、交易、账务领域业务规则相关，需人工进一步分析（额度视图、个人卡账户实际可用额度查询、e 招贷占用明细查询等工具）
- **相邻交易不连续**：前序交易实际可用额度后值与后序交易实际可用额度前值不一致时，需人工进一步分析（汇率变更历史查询、客户临/固额变更历史查询等工具）
- **特殊事件**：境外交易缴款购汇、国际卡组织交易退货（无授权直接入账）→ 需定制为处理事件的专项技能
- **兜底判定**：如业务归因仍不明确，则判定为系统异常，通知开发人员排障

**证据：**
> r1.md 中，客户预期 182344.58 - 42 = 182302.58，实际为 178501.06，差额 3843.52 由溢缴款规则引起：
>
> 1. 非循环额度节点还款（还款交易）不恢复额度
> 2. tcl005 非循环额度节点，溢缴款余额不影响可用额度
>
> 排障 9 步本质是逐步还原 f 的每一个参数。

**置信度：** 高

**Agent 含义：**
额度排障 Agent 的核心能力 = 给定交易区间，开展每笔交易可用额度连续性分析，实现区间内额度变化可解释。重点标记客户疑问或对客户感知不一致交易，并分析偏差项业务归因。

### F2：通过 Agent 提升排障效率依赖

**依赖项：**

1. 对存量数据查询接口工具识别定义（让 Agent 知道能做什么）
2. 将额度、交易、账务领域业务规则与配套工具封装为可复用的技能（让 Agent 理解应该怎么做）

**发现：**

排障需人工调用多个接口组合分析，对复杂问题还需要人工计算核对，按时序还原客户交易场景。此外还需综合额度、交易、账务领域业务规则来解释数据。

例如推理链：

```
入账交易码（4029/4306）→ 业务含义 → 额度恢复规则 → 溢缴款判定
```

**证据：**

- r1.md 步骤 2 查额度使用明细，步骤 5 切换到 CCA 运营管理查 4306 交易详情，涉及多接口输出组合分析、人工计算核对
- r1.md 步骤 4/6/7 三步均为「检索知识库匹配额度变化规则」，占排障流程 9 步中的 3 步（33%）。业务规则文档存在但分散不完整，依赖经验

**置信度：** 高

**Agent 含义：**

| 方向 | 说明 |
|------|------|
| **工具及技能封装** | 规范存量接口工具定义（规范入参、出参），设计存量多接口编排（工具间数据依赖），封装可复用技能（业务领域知识 + 接口工具），确保 Agent 输出稳定、可靠结论 |
| **RAG 知识库使用** | 业务规则，额度、交易、账务领域知识库。待明确：触发什么条件时 Agent 启用向量检索？启用向量检索用于什么？相比封装至 SKILL 的文字描述，独立 RAG 知识库的核心价值？ |

### F5：一期聚焦二线是正确的范围控制策略

**发现：**

二线客服已有排障能力，Agent 定位是「加速器」而非「替代者」。这意味着 Agent 输出错误时，二线可以识别并纠正，形成天然的 human-in-the-loop。一线客服缺乏这个兜底能力。

**证据：**
> 业务方明确提出「一期聚焦二线，成熟后推广一线」

**置信度：** 高

**Agent 含义：**

- **一期**：聚焦幻觉零容忍（自动验证）、典型单一场景、用户标记反馈
- **二期**：扩展场景、增加复合场景
- **三期**：面向一线
