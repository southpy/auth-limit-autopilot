# Harness 四象限需求评估报告：信用卡可用额度异常排障

## 1. 业务场景打分与定位

| 维度 | 分数 (1-10) | 核心理由 |
| :--- | :--- | :--- |
| **任务清晰度 (Task Clarity)** | **7** | 流程(SOP)明确，但依赖大量隐性业务知识（如特定交易码 4029/4306 的额度恢复逻辑）。 |
| **验证自动化程度 (Eval Automation)** | **9** | 额度变化是纯数学逻辑。公式 `可用额度变化 = f(交易金额, 业务规则, 系统参数)` 理论上可完全通过对账验证。 |

### 🎯 象限定位：**Autopilot (自动驾驶区)**
*注：该场景处于 Autopilot 区，非常适合 Agent 落地。验证环节的极高确定性抵消了业务规则匹配时的复杂性风险。*

---

## 2. 风险诊断 (Risk Diagnosis)

*   **知识断裂风险 (Knowledge Gap)**：SOP 中提到的业务规则（如“专享分期卡还款不恢复额度”）若未被结构化进入技能（Skill）或 RAG，Agent 会产生逻辑偏差。
*   **工具碎片化 (Tool Fragmentation)**：需要跨多个系统（额度明细、CCA 运管、汇率历史等）查询，Agent 的多工具链编排稳定性是关键。
*   **计算漂移 (Calculation Drift)**：对于溢缴款等涉及多步抵扣的场景，Agent 需要极其精确的数值处理能力。

---

## 3. 行动建议 (Action Plan)

1.  **MVP 阶段 (Autopilot 核心)**：
    *   **建立「额度对账审计」Skill**：将 `r1.md` 中的第 8 步（计算验证）固化。只要 Agent 给出的解释能推导出实际额度变动值，即判定为成功。
    *   **结构化交易码规则库**：不要只靠 RAG 搜文档，应将高频交易码（4029, 4306 等）及其额度影响逻辑封装为结构化数据或专门的查询 Skill。
2.  **二期阶段 (增强鲁棒性)**：
    *   **引入异常自查逻辑**：当 `f(params) != 实际变化` 时，Agent 应自动触发「深度探测」模式，遍历汇率、临额过期、退货授权等隐性影响因素。
3.  **Harness 设计建议**：
    *   **Gold Dataset**：收集 50+ 个已结案的复杂排障案例作为测试集，用「计算验证一致性」作为核心评估指标。
