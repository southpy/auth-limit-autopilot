# Agentic Engineering 全流程契约表 (SOP Contract Matrix)

*版本: v1.0 | 核心原则: 评估驱动开发 (EDD)、第一性原理、双向契约强制对齐*

| 阶段 (Phase) | 前置输入 (Inputs) | 物理交付物 & 退出契约 (Deliverables & Exit Criteria) | 场景举例 (异常退款 Agent) | 核心工具链推荐 (Toolchain) |
| :--- | :--- | :--- | :--- | :--- |
| **P0-P2 场景与评估** | 业务线原始痛点；未经清洗的客服工单/聊天记录库。 | **交付**：`harness_score.md` 评估表；`business_samples.jsonl` (≥30条清洗后的代表性业务语料)。 **契约**：明确落在 Harness 四象限的 **Q1 区**（任务高清晰、验证易自动化），且必须显式交付高质量真实语料样本，否则项目直接熔断。 | 抽取 50 条真实的“多扣款投诉”语料。判定退款政策明确，API 状态可机器断言。清洗出 30 条高质量语料进入下一阶段。 | **Claude Cowork**: 使用 `Agentic architect` 插件异步处理长文本，执行去熵与四象限评估。 |
| **P3a/3b 设计迭代环** | P2 产出的 Q1 场景认定；核心业务 SOP；≥30条真实语料。 | **交付**：`tool_schema.json`、`eval_criteria.py`、PRD（含 Case 分级标签 `critical/edge`，且 `critical` 占比 ≥ 50%）。 **契约**：Acceptance Criteria 与 Tool Schema 必须 **100% 双向映射**。即每个 AC 都有 Schema 字段支撑，且 Schema 的每个输出字段至少被一条 AC 校验，杜绝验证盲区。 | PRD 明确 `critical` 验收标准：“订单 > 30 天必须转人工”。该标准双向映射至 `execute_refund` 工具的 Error Envelope 中的 `is_retryable` 与 `retry_hint` 字段。 | **Claude Cowork**: 使用 `write-spec` 插件画 DAG 图； **Claude Chat (Pro)**: 快速脑暴 Schema 字段。 |
| **P4 MCP 封装** | P3a 锁定的 `tool_schema.json` 契约。 | **交付**：MCP Server 代码或本地 Tool Wrapper。 **契约**：所有涉及写操作的工具必须实现 Mock 拦截；必须抛出全局统一的结构化 Error Envelope。 | 封装内部支付网关。拦截对 `/api/refund` 的真实 POST 请求，在沙盒环境内永远返回 `{"status": "MOCK_SUCCESS"}`。 | **Cursor / Codex**: 编写底层 API 对接代码； **Claude Code**: 终端一键生成 MCP Wrapper 模板。 |
| **V0 & SanityCheck** | P4 封装好的 Tools；P0-P2 交付的 ≥30 条真实语料。 | **交付**：`v0_traces.jsonl`（包含人类与机器的对齐打分表）。 **契约**：人工打分与 Eval 自动化打分的 **一致率绝对值 ≥ 85%**。达不到则退回 P3b 重写 Eval 判卷逻辑，严禁修改 Agent 代码。 | V0 Agent 用真语料跑出 30 条轨迹。人工判定 Trace #12 违规（应转人工），但 Eval 脚本误判为 Pass。打回修改 Eval 脚本，直到机器与人对齐。 | **Claude Chat / 纯 Python**: 写最糙的单体 Prompt 跑出 Trace； **Jupyter Notebook**: 对比人工与机器打分对齐率。 |
| **P5 Eval 基建** | 对齐率达标的 Eval 脚本；V0 沉淀的 30 条 Trace。 | **交付**：`gold_dataset.jsonl` (从 30 条扩充至 100+ 条)；CI 评测脚本。 **契约**：执行 `make run_eval` 必须一键拉起流水线无报错；泛化生成的 Case **必须经人工 Review 确认**，严禁未审核的纯合成数据直接入库。 | 将 30 条 Trace 泛化为 150 条覆盖并发、超时等边缘场景的测试集，由人类专家审核后入库。集成 LLM-as-a-Judge 判定客服语气合规性。 | **Claude Code**: 自动生成 Pytest 测试桩代码； **Braintrust / Promptfoo**: 固化测试集与断言管线。 |
| **P6 生产级编码** | P3a 的架构拓扑；P5 完备的自动化测试网。 | **交付**：Agent 主干代码（基于 P3a 决定的原生 Tool Use 编排或轻量级状态机）。 **契约**：所有大模型调用必须挂载 `trace_id` 并在可观测面板成功落库可见。 | 编写退款 Agent 路由。在节点流转和 Tool 调用外层包裹 OpenTelemetry SDK 进行日志打点。克制引入重型多 Agent 框架，优先原生 API 编排。 | **Cursor / Codex**: 复杂工程级代码编排； **Langfuse / Phoenix**: LLM 原生监控面板接入。 |
| **P7 离线调优** | P6 产出的 V1 Agent 代码；本地运行的 Eval 基建。 | **交付**：Pass@K 性能面板截图。 **契约**：`critical` 主流程 Pass@K **≥ 95%**，`edge` 边缘流程 ≥ 85%。未达标打回 DesignLoop 修改契约，禁止盲目调优 Prompt。 | 运行 `make run_eval` 发现 Pass 率仅 80%。排查发现新增 Tool 导致上下文截断。回到 P3a 修改 Tool Schema 精简描述，而非在 Prompt 中打补丁。 | **Claude Code**: 在终端读取报错日志，高频自动化修改 System Prompt 并重新触发测试。 |
| **P8 渐进上线** | P7 达标的稳定版本 Agent。 | **交付**：流量分发网关配置；HITL 操作台。 **契约**：Shadow 期机器与人工的**最终决策一致率 > 95%**（不强求中间推理步骤一致）；HITL 期人工驳回率呈收敛趋势。 | 生产环境镜像 10% 客诉流量。Agent 生成工单草稿，人类客服在 Retool 后台点击“确认执行”才真实扣款。最终判定退款动作是否与人工决策一致。 | **LaunchDarkly / Envoy**: 流量路由与隔离； **Retool**: 快速搭建 HITL 人类审批后台。 |
| **P9 持续运营** | 真实生产环境（GA）的监控数据。 | **交付**：Bad Case 飞轮队列。 **契约**：发生数据漂移（Data Drift）则回注 P5 追加数据样本；发生规则盲区（Requirement Drift）则回流 P3b 重设规则。 | 用户使用未记录的海外信用卡支付导致退款失败。Agent 卡死。将此场景判定为“规则盲区”，回流 P3b 重新设计海外退款路由分支及验收标准。 | **Langfuse (Dashboard)**: 监控置信度异动； **Claude Cowork**: 异步批量分析 1000 个报错 Trace，提炼系统性失效模式。 |
